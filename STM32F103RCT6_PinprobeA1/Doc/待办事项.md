### 多机 FreeRTOS / CubeMX / NVIC 配置待办

| # | 内容 | 现状 / 修改建议 |
|---|------|----------------|
| M-RTOS.1 | 增加 CAN 接收任务 `CANRxTask` | 当前只有 `defaultTask`、`SCPI`、`ModBus`、`WatchDog`、`StateVector`；CAN RX 应由中断只入队，任务上下文解析协议 |
| M-RTOS.2 | 增加 CAN 发送任务 `CANTxTask` | CAN 发送不要在业务任务或 ISR 中阻塞；统一通过 TX queue 串行发送，便于重发、ACK、总线错误恢复 |
| M-RTOS.3 | 增加 CAN 协议/同步任务 `CANProtocolTask` 或合并到 `CANRxTask` | 处理心跳、入网、ACK、远程命令注入、IO 状态同步；不得直接调用 BsmRelay 硬件写接口 |
| M-RTOS.4 | 增加 FreeRTOS Queue / EventFlags | 需要 `CanRxQueue`、`CanTxQueue`、可选 `CanEventFlags`；HAL CAN 回调只投递帧或置位事件 |
| M-RTOS.5 | 复核任务栈和 heap | `configTOTAL_HEAP_SIZE=20480`，新增 CAN 任务后需重新估算；建议先预留 CANRx 512 words、CANTx 512 words、Protocol 512~768 words |
| M-RTOS.6 | 调整 `SysTimerCallback` 节拍分发 | 当前 25ms 只唤醒 `ModBus` 和 `StateVector`；多机需增加 CAN 周期事件，例如 100ms 心跳、500ms 离线检查 |
| M-RTOS.7 | `RamVector_Init(0)` 改为配置化 node_id | 当前 `StateVectorTask` 固定单机模式；多机时从 Flash / DIP / SCPI 配置读取 `node_id`，`0=单机`，`1~8=多机` |
| M-RTOS.8 | 明确 ModBusTask 与 CAN 同步边界 | 当前 `ModBusTask` 独占 RS485 并执行 `CmdExec_ExecuteAll()`；远程命令应写入 RamVector 命令槽，仍由本地执行链处理 |
| M-CUBE.1 | 定稿 CAN 引脚 | 当前 CubeMX 为 `PA11/PA12`；旧多机方案写 `PB8/PB9 Remap`。需要结合 PCB 确认后统一 `.ioc`、`can.c`、文档 |
| M-CUBE.2 | 定稿 CAN 波特率 | 当前 `.ioc` 计算约 428571bps (`Prescaler=6`, `BS1=12TQ`, `BS2=1TQ`)；多机建议统一 500kbps 或 1Mbps，并做实测 |
| M-CUBE.3 | 配置 CAN Filter 和启动流程 | `MX_CAN_Init()` 目前只初始化外设；后续需配置过滤器、`HAL_CAN_Start()`、激活 RX FIFO pending / error / bus-off 通知 |
| M-CUBE.4 | 复核 SPI Flash 外设选择 | 当前使用 `SPI3 PB3/PB4/PB5 + PC13 CS`；旧方案写 `SPI2 PB12~PB15`。W25Q128 阶段前必须统一硬件和 CubeMX 配置 |
| M-CUBE.5 | USART3 角色切换规划 | 当前 USART3 用于 RS485 IO 板通信；多机终态若要作为 PLC Modbus RTU 从站，需要确认是否与现有 IO 板 RS485 冲突 |
| M-CUBE.6 | 保持 SYS timebase 使用 TIM5 | 当前 FreeRTOS 下 HAL timebase 为 `TIM5_IRQn`，SysTick 留给 RTOS；该配置适合多任务，后续不要退回 SysTick timebase |
| M-NVIC.1 | 复核 CAN/DMA/SPI 中断优先级 | 当前 CAN、DMA、SPI 多为 preempt priority 5，USART 为 6，符合 `configLIBRARY_MAX_SYSCALL_INTERRUPT_PRIORITY=5`；若 ISR 调用 FreeRTOS FromISR API，优先级不得高于 5 |
| M-NVIC.2 | CAN ISR 只做轻量处理 | `USB_HP_CAN1_TX`、`USB_LP_CAN1_RX0`、`CAN1_RX1`、`CAN1_SCE` 已进 `HAL_CAN_IRQHandler()`；HAL 回调中只入队，不做协议解析和硬件执行 |
| M-NVIC.3 | 增加 CAN 错误/Bus-Off 诊断 | 使用 `CAN1_SCE_IRQn` 记录 bus-off、error passive、TX fail；错误事件投递到 CANProtocolTask，不在 ISR 里恢复复杂状态 |
| M-NVIC.4 | 评估 SPI DMA 与 UART DMA 优先级 | 后期 W25Q128 快照/日志会使用 SPI DMA，不能影响 USART RS485 与 CAN 实时性；必要时降低快照任务优先级并限制单次写入长度 |
| M-NVIC.5 | 避免中断中调用阻塞 HAL API | CAN、USART IDLE、SPI DMA 回调中禁止 `HAL_Delay`、阻塞发送、整表 memcpy、Flash 写入等耗时动作 |

### 多机线程资源推荐配置

> CubeMX 的 FreeRTOS Task `Stack Size` 按 word 填写；当前生成代码中表现为 `stack_size = N * 4` 字节。

| 任务 | CubeMX Stack Size | 生成代码字节数 | 优先级 | 调整理由 |
|------|-------------------|----------------|--------|----------|
| `defaultTask` | 128 | `128 * 4` | `osPriorityLow` | 仅做健康监控/后台占位，不应和实时任务抢占 |
| `SCPI` | 1024 | `1024 * 4` | `osPriorityLow` | SCPI 解析栈占用相对较大，保留当前余量；调试接口低优先级即可 |
| `ModBus` | 512 | `512 * 4` | `osPriorityNormal2` | 当前实际只是 RS485 IO 轮询、更新 RamVector、执行 CmdExec；局部缓冲很小，512 words 足够，若后续加入 PLC Modbus RTU 从站解析再升到 768 |
| `StateVector` | 768 | `768 * 4` | `osPriorityNormal1` | 状态机逻辑分支多，给出比当前 512 words 更稳的余量；低于 ModBus，避免阻塞 IO 执行 |
| `CANRxTask` | 512 | `512 * 4` | `osPriorityNormal2` | 及时消费 CAN RX queue，避免 FIFO/队列堆积；只做轻解析和投递事件 |
| `CANProtocolTask` | 768 | `768 * 4` | `osPriorityNormal1` | 处理心跳、入网、ACK、远程命令注入、IO 镜像同步；与 StateVector 同级 |
| `CANTxTask` | 512 | `512 * 4` | `osPriorityNormal` | CAN 发送可排队，不抢本机 IO；负责 mailbox、短超时、重发 |
| `WatchDog` | 128 | `128 * 4` | `osPriorityLow` | 仅喂狗，保持小栈；后续可加入任务心跳检查但不做复杂逻辑 |
| `SnapshotTask` | 512 | `512 * 4` | `osPriorityLow` | 后期 W25Q128 快照/日志再启用；必须低优先级、分片写入 |

建议优先级顺序：

```text
Normal2 : ModBus, CANRxTask
Normal1 : StateVector, CANProtocolTask
Normal  : CANTxTask
Low     : SCPI, WatchDog, defaultTask, SnapshotTask
```

建议 heap：

| 阶段 | `configTOTAL_HEAP_SIZE` | 说明 |
|------|-------------------------|------|
| 当前单机 | `20480` | 当前配置可继续使用 |
| 多机阶段 1~3 | `28672` | 新增 CANRx/CANTx/CANProtocol、CAN 队列和事件组后建议提高 |
| 加入快照/日志 | `32768` 起评估 | W25Q128 阶段再结合 map 文件和水位统计确认 |

上线前必须用 `uxTaskGetStackHighWaterMark()` 实测各任务最低剩余栈，连续运行 CAN 压力测试、SCPI 查询、RS485 IO 轮询后再收敛栈大小。
