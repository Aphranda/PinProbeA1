## 🔴 高风险 (4项) — 必须修复

### H1. UART3 双缓冲初始化错误

| 项目 | 详情 |
|------|------|
| **文件** | `Core/Src/stm32f1xx_it.c:68` |
| **严重性** | 🔴 高 |
| **影响** | 首次RS485通讯数据丢失/损坏 |
| **修复难度** | 一行修复 |

**问题描述**:

UART3 接收双缓冲的两个指针初始都指向同一块内存，与 UART1 的初始化不对称：

```c
// ✅ UART1 — 正确
uint8_t* Uart1_BuffIsReady  = Usart1_RX_BUF2;    // 指向 BUF2
uint8_t* Uart1_BuffOccupied = Usart1_RX_BUF1;    // 指向 BUF1  ← 不同!

// ❌ UART3 — 错误!
uint8_t* Uart3_BuffIsReady  = Usart3_RX_BUF2;    // 指向 BUF2
uint8_t* Uart3_BuffOccupied = Usart3_RX_BUF2;    // 也指向 BUF2! ← 相同!
```

**后果**: 首次 DMA `HAL_UART_Receive_DMA` 写入 `Uart3_BuffOccupied` (BUF2)，同时 `Uart3_BuffIsReady` 也指向 BUF2。第一次 IDLE 中断触发后指针才被交换分离。在 `main()` 的 POST 自检阶段就调用 `IO_Read()`，此时如果 RS485 从机响应，从 `Uart3_BuffIsReady` (实际上是还没有被正确交换的 BUF2) 读取数据，可能读到DMA正在写入的半帧数据。

**修复**:
```c
// 改为:
uint8_t* Uart3_BuffOccupied = Usart3_RX_BUF1;  // 与 UART1 一致
```

### H2. SCPI 任务与 ISR 之间的 TOCTOU 竞态条件

| 项目 | 详情 |
|------|------|
| **文件** | `Core/Src/main.c:317-332`, `Core/Src/stm32f1xx_it.c:349-380` |
| **严重性** | 🔴 高 |
| **影响** | SCPI命令解析错误、缓冲区数据混叠 |
| **修复难度** | 低-中等 |

**问题描述**:

```c
// SCPITask (main.c)
for(;;) {
    if (Uart1_RxLength > 3)                    // ← ① 检查
    {
        Uart1_BuffIsReady[Uart1_RxLength] = '\0'; // ← ② 写入
        SCPI_Input(&scpi_context, ...);            // ← ③ 解析
        Uart1_RxLength = 0;                        // ← ④ 清零
    }
    osDelay(10);
}

// USART1_IRQHandler (ISR) — 随时可打断上面的代码:
if (IDLE flag) {
    Uart1_RxLength = data_length;     // 修改长度
    // 交换双缓冲指针
    Uart1_BuffIsReady = ...;          // 修改指针
    // 重启 DMA 接收
    HAL_UART_Receive_DMA(..., Uart1_BuffOccupied, ...); // 开始覆盖旧缓冲
}
```

TOCTOU 攻击窗口：
- ① 检查通过 → ISR 触发新帧 → `Uart1_BuffIsReady` 被交换 + `Uart1_RxLength` 被覆盖 → ②③ 处理的是**旧指针+新数据混叠**或**空帧**

**修复建议**:
```c
for(;;) {
    uint32_t len;
    char *buf;
    taskENTER_CRITICAL();           // 关中断
    if (Uart1_RxLength > 3) {
        len = Uart1_RxLength;
        buf = (char *)Uart1_BuffIsReady;
        Uart1_RxLength = 0;
    } else {
        len = 0;
        buf = NULL;
    }
    taskEXIT_CRITICAL();            // 开中断

    if (buf && len > 0) {
        buf[len] = '\0';
        SCPI_Input(&scpi_context, buf, len);
    }
    osDelay(10);
}
```

### H3. 双状态机残留 — 旧代码未移除

| 项目 | 详情 |
|------|------|
| **文件** | `Hardware/BsmRelay/Src/StateMachine.c`, `Hardware/RamVector/Src/state_vector.c` |
| **严重性** | 🔴 高 |
| **影响** | RAM/Flash浪费、维护混淆、全局变量污染 |
| **修复难度** | 高 (需确认所有依赖后移除) |

**问题描述**:

项目存在**两套状态机实现**：

| 特性 | 旧状态机 (StateMachine.c) | 新状态机 (state_vector.c) |
|------|--------------------------|--------------------------|
| 全局变量 | `system_status` (全局), `lock_press_tick` (全局) | `system_status` (static本地), `lock_press_tick` (static本地) |
| 调用者 | **无** (未在main.c中调用) | `StateVectorTask` → `StateVector_Input()` |
| IO 来源 | 自己读 RS485 (`IO_Read`) | 从 RamVector 读 (`RamVector_GetLocalIO`) |
| 命令输出 | 直接调 `Cylinder_Write/Lock_Write` | 写 `RamVector.PostCmd` → `CmdExec` 执行 |
| 编译状态 | **仍编译链接，占用资源** | **实际运行中** |

`BsmRelay.c:13` 声明了 `extern uint8_t system_status;`，它链接到旧状态机的全局变量——但这个变量**从未被更新**（因为旧状态机不再运行），导致 `SYS_Status()` 返回的值永远停留在 `SYS_INIT`。

同时 `BsmRelay.c` 中的 `InputIO_Read()`、`OutputIO_Read()`、`Cylinder_Status()`、`Lock_Status()` 等函数通过 `IO_Read()` 直接操作 RS485 总线（参见 H2），它们可能在新状态机/SCPI路径中仍被间接调用。

**建议**:
1. 确认 `BsmRelay.c` 中哪些函数仍被使用 → 保留
2. 确认 `StateMachine.c` 不再被任何路径调用 → 从编译中移除
3. 将 `BsmRelay.c` 中的 `extern uint8_t system_status` 改为使用 `RamVector_GetState()`

## 🟡 中风险 (5项) — 建议修复

### M1. Flash 写入期间 CPU 暂停 → 看门狗风险

| 项目 | 详情 |
|------|------|
| **文件** | `Hardware/Flash/Src/flash.c` |
| **严重性** | 🟡 中 |
| **影响** | Flash保存时可能触发IWDG复位 |
| **修复难度** | 低 |

STM32F103 在 Flash 页擦除 (~40ms) 和编程期间**完全暂停 CPU 取指**。`Flash_Save()` 流程：

```
HAL_FLASH_Unlock()
Flash_ErasePage()        ← CPU暂停 ~40ms
HAL_FLASH_Lock()
HAL_FLASH_Unlock()
Flash_WriteData()        ← 逐双字编程, CPU反复暂停
HAL_FLASH_Lock()
```

IWDG 配置: `40000Hz / 64 / 2500 ≈ 4 秒`。单次 `Flash_Save` 约需 50-80ms，远小于 4s。但如果：
- 短时间内连续多次调用（如 SCPI 连续设置 IDN1~IDN4）
- 其他 ISR 未及时喂狗的极端情况

建议在擦除前主动喂狗。

**修复**:
```c
static Flash_Status_t Flash_ErasePage(void)
{
    HAL_IWDG_Refresh(&hiwdg);  // ← 擦除前喂狗
    HAL_FLASH_Unlock();
    // ...
}
```

### M2. Uart1_Printf / Uart3_Printf — TX Flag 死锁风险

| 项目 | 详情 |
|------|------|
| **文件** | `Core/Src/stm32f1xx_it.c:474-498` |
| **严重性** | 🟡 中 |
| **影响** | DMA异常后日志静默丢失，调试困难 |
| **修复难度** | 中等 |

```c
void Uart1_Printf(char *format, ...)
{
    if (Usart1_TX_Flag) return;  // TX忙 → 静默丢弃本帧!
    // ...
    Usart1_TX_Flag = 1;
    HAL_UART_Transmit_DMA(&huart1, ...);
}

// DMA TC 中断中:
Usart1_TX_Flag = 0;  // 仅在DMA完成时清零
```

问题：
1. **DMA 失败 = flag 永久卡死**: 如果 DMA 传输因总线错误/噪声失败，TC 中断不触发 → flag 永远为 1 → 所有日志静默丢失
2. **TX_BUF 共享无保护**: `Usart1_TX_BUF` 在 `vsnprintf` 阶段若被另一线程同时写入 → 数据错乱
3. **静默丢帧**: TX忙时直接return，没有队列缓冲，关键错误信息可能被丢弃

**建议**: 引入带超时的重试机制 + DMA错误回调重置flag:
```c
void Uart1_Printf(char *format, ...) {
    uint32_t waited = 0;
    while (Usart1_TX_Flag && waited < 100) {  // 最多等100ms
        osDelay(1); waited++;
    }
    if (Usart1_TX_Flag) return;  // 确实超时了才丢

    taskENTER_CRITICAL();  // 保护 vsnprintf + 发送
    vsnprintf(...);
    Usart1_TX_Flag = 1;
    HAL_UART_Transmit_DMA(...);
    taskEXIT_CRITICAL();
}
```

或者在 DMA 错误中断中重置 flag。

### M3. DMA 中断中直接修改 HAL 内部状态

| 项目 | 详情 |
|------|------|
| **文件** | `Core/Src/stm32f1xx_it.c:201-274` |
| **严重性** | 🟡 中 |
| **影响** | HAL状态机不一致 |
| **修复难度** | 低-中等 |

```c
void DMA1_Channel2_IRQHandler(void)
{
    // ⚠️ 在 HAL_DMA_IRQHandler 之前手动修改 HAL 内部状态
    if(__HAL_DMA_GET_FLAG(&hdma_usart3_tx, DMA_FLAG_TC2) != RESET)
    {
        __HAL_UART_CLEAR_IDLEFLAG(&huart3);
        Usart3_TX_Flag = 0;                 // ✅ 自己的变量, OK
        huart3.gState = HAL_UART_STATE_READY;  // ⚠️ HAL内部变量!
        hdma_usart3_tx.State = HAL_DMA_STATE_READY;  // ⚠️ HAL内部变量!
    }
    HAL_DMA_IRQHandler(&hdma_usart3_tx);  // HAL可能再次修改这些状态
}
```

在 HAL 的 `HAL_DMA_IRQHandler` 调用前手动修改 `huart3.gState` 和 `hdma_usart3_tx.State`，HAL 内部状态机可能产生非预期转换。

**建议**: 改用 HAL 回调 `HAL_UART_TxCpltCallback()` 来设置 `Usart3_TX_Flag = 0`，不要手动干预 HAL 状态结构。

### M4. IOWriteOrder / IOReadOrder 返回静态缓冲区

| 项目 | 详情 |
|------|------|
| **文件** | `Hardware/modbus/src/RS485.c:51-117` |
| **严重性** | 🟡 中 |
| **影响** | 潜在数据覆盖（当前使用方式恰好安全） |
| **修复难度** | 低 |

```c
uint8_t* IOWriteOrder(uint8_t index, uint8_t status) {
    static uint8_t stress_send[9];  // ⚠️ 静态缓冲
    // ... 填充 stress_send
    return stress_send;
}

uint8_t* IOReadOrder(uint8_t func, uint16_t num) {
    static uint8_t stress_send[9];  // ⚠️ 共享同一块静态内存!
    // ... 填充 stress_send
    return stress_send;
}
```

两个函数都返回指向各自 static 数组的指针。当前调用方在拿到指针后立即使用，所以安全。但 `stress_send[9]` 对于 ModBus 帧（8字节 + 终止符）刚好够，容错余量不足。如果将来添加第9字节，就是缓冲区溢出。

**建议**: 改为调用方提供缓冲区，或使用 `static` + 注释强制说明"调用后立即使用"。

### M5. SCPI 波特率配置命令阻塞无超时

| 项目 | 详情 |
|------|------|
| **文件** | `Hardware/libscpi/port/scpi-def.c:267-323` |
| **严重性** | 🟡 中 |
| **影响** | BSM从机无响应时命令永久阻塞 |
| **修复难度** | 低 |

`SCPI_Configurebaudrate()` 配置 BSM 从机波特率时使用了 `HAL_MAX_DELAY`：

```c
HAL_UART_Transmit(&huart3, set_baud_cmd, sizeof(set_baud_cmd), HAL_MAX_DELAY);
HAL_Delay(100);
HAL_UART_Transmit(&huart3, write_flash_cmd, sizeof(write_flash_cmd), HAL_MAX_DELAY);
// ...
HAL_UART_Receive(&huart3, response, sizeof(response), 100);  // 100ms超时
```

如果 BSM 从机不在线或故障，前几个 `HAL_MAX_DELAY` 调用可能导致**永久阻塞**（取决于HAL TX超时配置，在阻塞模式下可以是无限等待）。

**建议**: 将 `HAL_MAX_DELAY` 替换为明确的超时值（如 500ms），并检查返回值。

## 🟢 低风险 / 改进建议 (7项)

### L1. 事件表 (event_table) 未集成使用

| 文件 | `Hardware/RamVector/Src/event_table.c` |
|------|------|
| **说明** | `EventTable_Lookup()` 已实现，但在 `state_vector.c` 中完全没有被调用。状态机使用内联 `if/else` 判断而非查表。 |
| **建议** | 要么将状态机改为查表驱动（提高可维护性），要么删除死代码。 |

### L2. CAN 总线硬件初始化但无处理任务

| 文件 | `Core/Src/can.c`, `stm32f1xx_it.c` |
|------|------|
| **说明** | CAN 外设被初始化、中断被使能，但无 FreeRTOS 任务消费 CAN 消息。多机架构仅停留在 RAM Vector 的数据结构预留层面。 |
| **建议** | 若当前无多机需求，在 CubeMX 中禁用 CAN 以节省 NVIC 资源和 RAM。 |

### L3. Fault Handler 无诊断信息输出

| 文件 | `Core/Src/stm32f1xx_it.c:106-176` |
|------|------|
| **说明** | HardFault、MemManage、BusFault、UsageFault 处理器全部是空的 `while(1)` 死循环。一旦触发，设备静默卡死，无法远程诊断。 |
| **建议** | 至少输出栈帧关键寄存器（PC、LR）到 UART1 辅助现场排查。 |

示例：
```c
void HardFault_Handler(void)
{
    // 在进入死循环前尝试输出诊断
    Uart1_Printf("[HARDFAULT] HFSR=0x%08X\r\n", SCB->HFSR);
    Uart1_Printf("[HARDFAULT] CFSR=0x%08X\r\n", SCB->CFSR);
    while(1);
}
```

### L4. ram_vector 固定地址依赖链接脚本

| 文件 | `Hardware/RamVector/Src/ram_vector.c:12` |
|------|------|
| **说明** | `__attribute__((section("RamVector"), aligned(4)))` 期望地址 `0x2000A000`。需要确认 `.ld` 链接脚本正确配置了 `.RamVector` section，否则变量可能落在 heap/stack 重叠区。 |
| **建议** | 在 `.ld` 文件中独立声明 section 并在 startup 中跳过初始化，或使用绝对地址指针 `#define RAM_VECTOR ((RAM_Vector_t*)0x2000A000)` 代替。 |

---

### L5. GetTim1Ms() 32位溢出（当前用法安全）

| 文件 | `Core/Src/tim.c:116-118` |
|------|------|
| **说明** | `tim1_ms` 是 32-bit 1ms 计数器，约 49.7 天溢出。当前代码均使用 `(now - start) >= threshold` 模式（unsigned wraparound 安全）。 |
| **建议** | 在代码规范中明确要求所有时间比较使用差值法，禁用绝对时间比较。 |

---

### L6. showDoorStatus() 缺少调试宏保护

| 文件 | `Hardware/BsmRelay/Src/StateMachine.c:642-658` |
|------|------|
| **说明** | `showStatus()` 有 `#ifdef Debug` 保护，但 `showDoorStatus()` 没有。如果旧状态机仍被编译，每次门状态变化都会无条件打印。 |
| **建议** | 给 `showDoorStatus()` 也加上 `#ifdef Debug`，或随旧状态机一起移除。 |

---

### L7. Flash_Config_t 版本兼容性严格

| 文件 | `Hardware/Flash/Src/flash.c:103-107` |
|------|------|
| **说明** | `Flash_VerifyConfig()` 要求主版本号**必须严格匹配**，否则视为无效配置（回退到默认值）。如果未来升级固件并增加配置字段，旧设备升级后配置会丢失。 |
| **建议** | 增加版本迁移函数，支持从旧版本结构体升级到新版本。 |
