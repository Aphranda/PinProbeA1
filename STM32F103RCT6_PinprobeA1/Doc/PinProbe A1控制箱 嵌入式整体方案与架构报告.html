项目目标与边界
当前阶段：单机箱体控制固件  |  后续阶段：CAN 多机协同
项目定位
  • 控制 PinProbe A1 箱体的气缸、锁、LED、传感器和安全输入
  • 对外提供 SCPI 调试/控制接口，内部通过 RS485 控制 BSM IO 板
  • 用 FreeRTOS 将通讯、状态机、看门狗和诊断任务分离
  • 以 RamVector 作为唯一状态与命令交换中心
当前已实现
  • 25ms 系统节拍驱动 ModBusTask 与 StateVectorTask
  • SCPI 命令写入 RamVector 命令槽，状态查询读取 IO 镜像
  • RS485 故障保护、门限位消抖、急停/激光防夹、风险模式
  • Flash 配置、POST 自检、固件版本/运行时间/日志查询
  • 单机 OTA 接收、W25Q128 暂存、Bootloader 搬运与 App 确认
明确不混入当前阶段
  • CAN 多机同步仍属于后续阶段，不视为单机已落地功能
  • W25Q128 快照、CAN OTA 跨节点分发、跨节点仲裁按多机方案统一规划
  • 波特率配置是出厂/维护动作，不作为运行期控制链路
  • 旧 StateMachine.c 已废弃，只保留历史参考
当前主线目标：把单机箱体控制做成 低耦合、可诊断、可扩展 的工业控制内核

链路段设计选择工程意义
控制入口SCPI 只负责参数解析和命令发布不会直接碰 RS485，避免和周期 IO 任务争总线

RamVector 黑板式数据总线
固定 1024B 内存布局  |  单机可用，多机预留  |  命令/状态/IO 解耦
区域内容
帧头magic、version、timestamp、crc32，为多机同步预留
命令区lock、cylinder、led 三个独立 CmdSlot，带优先级
路由区node_mask、source_node、ack_flags，多机阶段启用
状态区global_state、error_code、node_status[8]
IO 镜像io_state[8]，单机只用 index 0
扩展区ext_params[786]，给后续复杂参数和版本扩展留余量
架构价值
  • SCPI、状态机、执行层都围绕同一份数据表协作
  • 命令是意图，IO 镜像是事实，状态是推导结果
  • 单机阶段不需要复杂队列即可获得清晰模块边界
  • 多机阶段可以直接把同一张表扩展为 CAN 同步载体
已识别加固点
  • IO 镜像读取需要整帧快照，避免字段半新半旧
  • 命令槽需要原子 Take/Clear，避免新命令被 Clear 覆盖
  • 执行命令仍保持在临界区外，避免阻塞实时路径
RamVector 是当前项目的核心边界：外部命令、现场 IO、系统状态 都在这里汇合

上电初始化链路
  • HAL、时钟、GPIO/DMA/UART/SPI/CAN/TIM/IWDG 初始化
  • 开启 UART IDLE 和 DMA 中断，启动 USART1/USART3 DMA 接收
  • Flash_Init 加载配置，SCPI_Init 建立命令上下文
  • AppLog_Init 后执行 POST：RCC、Flash CRC、RS485、IO板型号
  • 启动 FreeRTOS 调度器，由 SysTimer 驱动周期任务

诊断策略

项目没有把诊断混入 RamVector，而是用独立 AppLog 记录“发生了什么”。RamVector 负责实时状态，AppLog 负责事后解释，两者边界清晰。

AppLog 结构化诊断黑匣子
独立于 RamVector 的诊断面  |  固定 RAM 区  |  SCPI 拉取 + 实时 UART
设计点实现方式价值
独立内存段scatter 固定 RW_APPLOG 0x2000A400，大小 0x500诊断数据不占用 RamVector，避免污染实时状态表
固定记录格式每条 16B，含 tick、seq、node、level、module、event、arg比 printf 字符串更省 RAM，也便于后续落盘或上传
环形缓冲容量 76 条，满后覆盖最旧记录并累计 drop_count不会因日志堆积阻塞控制链路
双消费通道SCPI read_index 与实时 realtime_index 分离人工查询和实时串口输出互不抢同一个游标
文本格式化读取时再格式化为 STATE/ACTION/EVENT/IO 文本运行期存结构体，输出期再转换，节省现场 RAM
当前记录内容
  • 状态跳转：LOCK、IDLE、READY、RUNNING、EMERGENCY、COMPLETE
  • 动作耗时：关门开始/完成、开门开始/完成、阶段里程碑
  • 安全事件：RS485 故障、急停、激光防夹、风险模式、气压异常
  • 命令来源：SCPI 锁、气缸、LED 操作，以及按钮触发
  • IO 快照：按需记录输入/输出原始字节变化
访问方式
  • READ:LOG:NEXT? 拉取下一条日志
  • READ:LOG:ALL? 拉取当前缓存全部日志
  • READ:LOG:STATus? 查看容量、待读数、丢弃计数、实时开关
  • CONFigure:LOG:UART 开关实时 UART 输出
  • UART 忙时实时输出直接跳过，不等待串口空闲

架构定位

RamVector 回答“系统现在是什么状态”，AppLog 回答“系统为什么变成这个状态”。它是单机调试、现场验收、异常复盘和后续 Flash 故障日志的共同基础。

注：实时 UART 是 best-effort 诊断输出，不承诺全量；需要完整读取时以 SCPI 拉取通道和 drop_count 为准。

设计点当前实现工程意义
内部 FlashBootloader 常驻前段，App 从 0x08006000 启动，配置页保持 0x0803F800App 不自擦写自身，升级动作由更小的 Bootloader 控制
W25Q128 分区OTA Area 从 0x015000 开始；Metadata A、Boot Flags、Control Log、Slot A/B 独立划分OTA 与快照、事件日志分区隔离，降低互相覆盖风险
传输协议SYSTem:OTA:BEGIN/DATA/END/VERify?/COMMit/ABORt/BOOT?，DATA 使用 #3xxx binary block复用现有 SCPI 维护口，同时避免把固件数据塞进 RamVector
完整性检查App 侧按块写 W25Q128 并计算整包 CRC32；Bootloader 再次校验 Slot CRC 和向量表传输完成和最终刷写前各校验一次，避免错误镜像进入 App 区
上位机工具Tools/pinprobe_gui.py 已集成 OTA 页，上传期间独占串口并暂停普通轮询升级流程有可操作入口，可显示进度、速度、Boot 状态和失败阶段
已落地模块
  • Hardware/OTA 定义 Manifest、Boot Flags、状态机和错误码
  • Bootloader/Src/boot_main.c 完成 Slot A 到 App 区的搬运与回读校验
  • scpi-def.c 暴露 OTA 状态、上传、校验、提交和中止命令
关键边界
  • App 只接收、暂存、校验和提交，不直接擦写运行中的 App 区
  • RamVector 不承载固件数据，只适合放 OTA 状态摘要和多机 ACK 位图
  • 提交后通过 SRAM BootRequest 和 W25Q128 Boot Flags 交接给 Bootloader
后续扩展
  • CAN 多机阶段由主节点分发已校验镜像，子节点仍本地校验后刷写
  • Slot B、失败回滚、签名校验和更大分块可按同一 Manifest 扩展
  • OTA 状态可映射到 RamVector 扩展区，供集群健康监控读取
OTA 的核心原则：App 管接收，W25Q128 管暂存，Bootloader 管刷写

核心优势总结
当前实现的工程价值  |  面向现场可靠运行
1. 低耦合控制内核
  • SCPI、状态机、RS485 执行层不直接互相驱动
  • RamVector 作为唯一共享数据平面
  • 命令、事实、状态分区清楚，便于审查和扩展
2. 确定性时序
  • 25ms 统一节拍驱动关键控制路径
  • ModBusTask 独占现场总线，避免串口竞争
  • 状态机不直接发 RS485，逻辑执行时间稳定
3. 安全优先级明确
  • 急停、激光、RS485 故障使用更高命令优先级
  • 断电回 LOCK、按钮确认、限位消抖形成多层保护
  • 风险模式可配置，不把兜底逻辑写死
4. 现场可诊断
  • POST 输出启动自检结果与复位原因
  • AppLog 记录状态跳转、动作耗时、关键事件
  • SCPI 可读取版本、运行时间、IO、日志状态
5. 配置可持久化
  • Flash 配置带魔数、版本和 CRC 校验
  • IDN、急停类型、风险模式可通过 SCPI 配置
  • 配置失效时可回默认值，不阻塞固件启动
6. 多机预留充分
  • RamVector 已按 8 节点布局预留 node_status 和 io_state
  • CAN、SPI 外设初始化已在工程中存在
  • defaultTask 已留出集群健康监控位置
7. OTA 维护闭环
  • App 侧接收固件并写入 W25Q128 Slot A，不自擦写运行区
  • Bootloader 校验 Manifest、CRC 和向量表后再搬运到 App 区
  • GUI 已提供固件选择、上传、校验、提交和 Boot 状态观测入口

阶段目标主要内容收益
阶段 1单机上线加固已完成气缸写入顺序、IO 快照、命令槽 Take/Clear、WriteIO/ReadIO 诊断降低现场偶发风险
阶段 2产线测试模式增加一键 IO/传感器/通讯自测，输出标准化验收结果，补充 SCPI 错误反馈覆盖缩短装配和维修定位时间
阶段 3运行数据沉淀将关键故障、动作计数、设备寿命累计迁移到 W25Q128 或备份区，并复用现有分区边界支持预防性维护
阶段 4OTA 可靠性增强完善 Slot B 回滚、掉电恢复、签名校验、分块严格校验和重复块读回比较提升现场升级成功率

最终结论
当前项目的技术判断  |  保持主架构，做边界加固
RamVector 作为主数据面ModBusTask 独占现场总线StateVector 保持纯逻辑SCPI 保持调试入口

总体判断

PinProbe A1 当前单机固件已经形成清晰的工业嵌入式控制框架：上位机命令、现场 IO、状态机、安全事件、配置与诊断分别有明确边界。项目的核心价值不在复杂抽象，而在用较少模块实现了可预测的控制闭环。

保留
  • 25ms 时间触发主节拍
  • RamVector 命令/状态/IO 模型
  • 状态机按物理语义顺序展开
加固
  • 执行器写入顺序
  • 共享数据快照一致性
  • 命令取清原子性和失败诊断
扩展
  • 出厂测试与验收自动化
  • W25Q128 日志/快照与 CAN OTA 分发
  • CAN 反射内存式多机协同
推荐路线：不重构主架构，沿当前边界逐项加固，再进入多机阶段
