# PinProbeA1 代码审查报告

日期: 2026-06-13  
范围: `Core`、`Hardware`、`.eide`、`MDK-ARM` 中的应用代码、RTOS 配置、链接脚本与构建配置。  
说明: 本报告基于静态代码审查，未修改源码，未执行完整固件编译和硬件联调。

> 二次复核说明: 初版报告中若干表述偏“风险清单化”，没有充分区分“完全没有保护”和“已有保护但保护边界不足”。本次复核重点补充已有保护、残余风险和优先级判断，避免把已经做对的设计误判为缺失。

## 总体结论

当前项目架构主线较清晰: STM32F103 + HAL + FreeRTOS，业务逻辑从旧 `StateMachine` 迁移到 `RamVector/state_vector/cmd_exec`。

最关键的设计点是中间向量表 `RamVector`: 箱体控制动作没有让 SCPI、按钮状态机直接读写执行器，而是通过 RAM 向量表做隔离。

- SCPI 命令只把用户意图写入 `RamVector` 命令槽。
- `state_vector.c` 只从 `RamVector` 读取 IO 镜像，并把按钮/安全事件转换为命令槽写入。
- `ModBusTask` 负责 RS485 IO 采样、更新 `RamVector` IO 镜像，并调用 `cmd_exec.c` 执行命令。
- `cmd_exec.c` 是从向量表命令到 BSM IO 写操作的执行边界。

这个隔离方向是正确的，能把“上位机协议解析”“状态机决策”“硬件执行”拆开，降低模块间直接耦合。本次审查中关于 `RamVector` 的风险不是否定该架构，而是指出向量表作为跨任务共享边界后，需要补齐原子性、一致性和初始化时序保护。

二次复核后需要特别说明: 当前代码已经有不少保护，不应按“裸奔”理解。

- `RamVector_Post*()` 已有三通道优先级仲裁，低优先级命令不会覆盖高优先级命令。
- `SCPITask()` 对 USART1 接收长度和缓冲指针做了临界区快照，避免 ISR 正在切换缓冲时被任务读到不一致的指针/长度组合。
- `state_vector.c` 已有 RS485 连续失败阈值，失败时会跳过本轮自动逻辑，并用 LED 命令告警。
- 急停、激光防夹命令使用 `CMD_PRIO_SAFETY`，且安全事件处理后跳过普通按钮事件。
- `ReadHoldingRegister()` 已经校验 7 字节长度、地址、功能码、字节数和 CRC；风险主要集中在 `IO_Read()` 的读输入/读输出路径。

因此，后文中 `RamVector` 和串口相关问题应理解为“已有局部保护，但跨任务事务边界仍不完整”，而不是主架构错误。

## 中间向量表架构评估

### 已经做对的部分

1. SCPI 命令没有直接调用 `WriteIO()` 控制门、锁、灯，而是调用 `RamVector_PostCylinder()`、`RamVector_PostLock()`、`RamVector_PostLED()` 写入意图。
2. 状态机 `state_vector.c` 不直接读 RS485，而是读取 `RamVector_GetLocalIO()` 提供的 IO 镜像。
3. RS485 读写集中在 ModBus/BSMRelay 执行侧，状态判断和硬件执行没有互相穿透。
4. `cmd_exec.c` 作为命令执行适配层，将向量表中的 `VCMD_*` 转换为 BSM IO 写操作，边界清楚。
5. 命令槽拆成 Lock、Cylinder、LED 三通道，并带优先级字段，能表达用户命令和安全命令的仲裁关系。

### 需要补强的边界

1. 向量表是共享内存边界，不只是数据结构。所有跨任务访问都应通过并发安全 API 完成。
2. `Post`、`Take`、`Clear` 必须形成原子语义。当前已有优先级仲裁，但没有把“读取并清空命令槽”做成一个不可被打断的事务。
3. IO 镜像更新和读取最好提供快照接口，避免状态机或 SCPI 查询读到更新到一半的结构体。
4. `RamVector_Init()` 应在调度器启动前完成，使向量表从系统启动起就是有效边界。
5. 少数旁路命令需要明确归类。例如 `CONFigure:BAUDrate` 直接操作 USART3，属于维护/配置类旁路，应有独占锁和暂停 ModBus 的机制。

主要风险集中在以下几类:

1. Flash 配置区与链接脚本 IROM 范围冲突，存在运行时擦写固件代码的风险。
2. RS485 接收帧长度校验不足，有明确越界读。
3. 中间向量表 `RamVector` 已承担隔离职责，且已有优先级仲裁，但命令槽缺少跨任务 take/clear 原子事务。
4. SCPI 选择参数本身多数由 `SCPI_ParamChoice()` 保护；风险主要来自硬件读回值、Flash 保存值、系统状态值作为数组下标前缺少二次范围校验。
5. UART1/USART3 的发送接收路径已有部分缓冲/临界区保护，但仍存在协议输出、日志输出、DMA 接收、阻塞接收之间的边界混用。
6. 构建配置之间不一致，存在引用不存在文件、旧构建缓存包含废弃文件的问题。

## 关键问题复核

### 1. Flash 配置页与 IROM 链接范围冲突

位置:

- `MDK-ARM/STM32F103RCT6_PinprobeA1.sct:1`
- `MDK-ARM/STM32F103RCT6_PinprobeA1.uvprojx:249`
- `Hardware/Flash/Inc/flash.h:39`
- `Hardware/Flash/Src/flash.c:311`

现象:

`flash.h` 将配置区放在最后一页 Flash:

- `FLASH_CONFIG_PAGE = 127`
- `FLASH_CONFIG_ADDR = 0x0803F800`
- 页大小 2KB

复核时发现 scatter 文件曾配置完整 256KB IROM:

```text
LR_IROM1 0x08000000 0x00040000
ER_IROM1 0x08000000 0x00040000
```

这表示代码链接区域覆盖完整 256KB Flash，包括 `0x0803F800..0x0803FFFF`。一旦固件体积增长并占用最后 2KB，运行时 `Flash_Save()` 擦除配置页会擦掉程序代码。

修复状态:

- `MDK-ARM/STM32F103RCT6_PinprobeA1.sct` 已将 `LR_IROM1/ER_IROM1` 长度改为 `0x0003F800`，即 `0x08000000..0x0803F7FF`。
- `.eide/eide.yml` 和 `.eide/eide-Aphranda.json` 的 ROM size 已改为 `0x3F800`，并指向同一份 scatter 文件。
- `MDK-ARM/STM32F103RCT6_PinprobeA1.uvprojx` 已启用 scatter 文件 `.\STM32F103RCT6_PinprobeA1.sct`，CPU IROM 描述同步为 `0x08000000..0x0803F7FF`。
- 剩余需要验证的是实际构建产物 map 文件，确认 `.text/.rodata` 未进入 `0x0803F800` 之后。

影响:

- 设备运行中执行配置保存后可能损坏固件。
- 故障表现可能是重启后 HardFault、无法启动、程序跑飞。
- 注释中提到“链接脚本已减少 2KB”，但当前 `.sct` 未体现。

建议:

- 将 IROM 长度改为 `0x0003F800`，显式保留最后 2KB 给配置页。
- 同步检查 `.eide/eide.yml`、Keil 工程、实际产物 map 文件，确保所有构建入口一致。
- 在链接后增加产物尺寸检查，禁止 `.text/.rodata` 进入 `0x0803F800` 之后。

优先级: 已修复，待构建验证

### 2. RS485 IO 读取存在越界读

位置:

- `Hardware/BsmRelay/Src/BsmRelay.c:158`
- `Hardware/BsmRelay/Src/BsmRelay.c:167`
- `Hardware/BsmRelay/Src/BsmRelay.c:168`
- `Hardware/BsmRelay/Src/BsmRelay.c:169`

现象:

`IO_Read()` 判断:

```c
if (Uart3_RxLength < 6)
```

但后续读取:

```c
memcpy(IOData, Uart3_BuffIsReady, 5);
crcData[0] = Uart3_BuffIsReady[5];
crcData[1] = Uart3_BuffIsReady[6];
```

访问到下标 6，最低需要 7 字节。当前长度为 6 的异常帧会越界读。

二次复核:

- 这个问题仍成立，且是明确代码级 bug。
- 但范围应限定在 `IO_Read()` 这条读输入/读输出路径。
- `Hardware/modbus/src/RS485.c` 中的 `ReadHoldingRegister()` 已经使用 `< 7` 长度判断，并校验地址、功能码、字节数和 CRC，不应把保持寄存器读取路径一并算作同类缺陷。

影响:

- CRC 判断使用未定义数据。
- 异常帧可能被误判为有效。
- 在某些编译优化/内存布局下可能引入难复现故障。

建议:

- 将长度判断改为 `< 7`。
- 同时校验响应帧地址、功能码、字节数。例如输入/输出读取预期为:
  - 地址 `0x01`
  - 功能码 `0x01` 或 `0x02`
  - 字节数 `0x02`
  - 总长度 7 字节
- 建议封装统一的 ModBus 响应解析函数，避免每个调用点手写长度和 CRC。

优先级: P0

### 3. RamVector 隔离架构正确，已有优先级仲裁，但缺少 take/clear 原子事务

位置:

- `Hardware/RamVector/Src/ram_vector.c:39`
- `Hardware/RamVector/Src/ram_vector.c:47`
- `Hardware/RamVector/Src/ram_vector.c:55`
- `Hardware/RamVector/Src/ram_vector.c:70`
- `Hardware/RamVector/Src/cmd_exec.c:64`
- `Hardware/RamVector/Src/cmd_exec.c:77`
- `Hardware/libscpi/port/scpi-def.c:431`
- `Hardware/libscpi/port/scpi-def.c:480`
- `Hardware/libscpi/port/scpi-def.c:510`
- `Hardware/RamVector/Src/state_vector.c:453`
- `Hardware/RamVector/Src/state_vector.c:488`

现象:

项目已经通过 `RamVector` 将控制意图和硬件执行隔离，这是正确的主架构。当前问题出在这个共享边界的并发语义还不完整。

命令槽由多个上下文写入:

- SCPI 任务写入用户命令。
- StateVector 任务写入按钮/安全命令。
- ModBus 任务读取并执行，然后统一 `RamVector_ClearCmd()`。

当前已有的保护:

- `cmd_lock`、`cmd_cylinder`、`cmd_led` 三个槽位彼此独立，互不覆盖。
- `RamVector_Post*()` 只有在 `prio >= current.priority` 时才更新槽位，能防止普通用户命令覆盖安全命令。
- `state_vector.c` 的急停/激光事件使用 `CMD_PRIO_SAFETY`，普通按钮/SCPI 命令使用 `CMD_PRIO_USER`。

这些保护是有效的，但它们只解决“写入时的优先级覆盖”问题，没有解决“消费者读取后统一清槽”与“生产者同时写入”的竞态。

当前 `Post/Get/Clear` 没有临界区、互斥锁或单生产者单消费者队列。也就是说，隔离层已经存在，优先级仲裁也已经存在，但隔离层内部的跨任务取命令事务还没有闭合。典型竞态:

1. ModBusTask 读取 lock/cylinder/led 命令。
2. SCPI 或 StateVector 写入新命令。
3. ModBusTask 调用 `RamVector_ClearCmd()`。
4. 新命令被清掉，永远不会执行。

影响:

- 用户命令偶发丢失。
- 安全命令理论上也可能受清槽时序影响，但由于 StateVector 周期性重发安全状态，实际风险低于普通单次 SCPI 命令丢失。
- 优先级仲裁在单次写入时成立，但无法保证跨任务 take/clear 原子性。
- 隔离架构本身仍然成立，但需要把 `RamVector` API 做成真正的并发安全边界。

建议:

- 最小修复: `RamVector_Post*()`、`RamVector_Get*()`、`RamVector_ClearCmd()` 进入 FreeRTOS 临界区。
- 更稳妥方案: 提供 `RamVector_TakeCommands()`，在一个临界区内复制并清空命令槽，让 ModBusTask 只调用这一种取命令接口。
- 如果需要保留“最新意图覆盖旧意图”的语义，继续使用三通道命令槽是合理的，不一定要改成队列。
- 如果未来需要保留每个命令历史，再考虑 FreeRTOS queue 或 message buffer。

优先级: P0/P1

说明: 若按“用户命令偶发丢失”评估可列 P1；若要求安全命令首帧必须一次命中、不能依赖周期性重发，则列 P0。当前代码已有安全优先级和周期性状态机保护，建议修复优先级放在 RS485 越界之后。

### 4. SCPI 与状态查询存在残余数组越界风险

位置:

- `Hardware/libscpi/port/scpi-def.c:372`
- `Hardware/libscpi/port/scpi-def.c:378`
- `Hardware/libscpi/port/scpi-def.c:396`
- `Hardware/BsmRelay/Src/BsmRelay.c:136`

现象:

`SCPI_ConfigureLink()` 和 `SCPI_ReadLinkState()` 直接使用 `param` 或 `mask` 访问 `link_source[]`:

```c
Link_Write(1, link_source[param].tag)
name_read = link_source[mask].name;
name = link_source[mask].name;
```

`SYS_Status()` 直接使用 `RamVector_GetState()` 访问 `sys_source[]`:

```c
return sys_source[RamVector_GetState()];
```

二次复核:

- `param` 来自 `SCPI_ParamChoice(context, link_source, &param, TRUE)`，正常 SCPI 输入路径已经受到 choice 表保护，不能简单说“SCPI 参数未校验”。
- `mask` 来自 `Link_Read()` 的硬件读回，仍需要范围校验。
- `RamVector_GetState()` 正常由 `state_vector.c` 设置，枚举范围为 `0..6`，但 `RamVector_Init()` 后默认 `global_state=0`，后续共享内存损坏或未来扩展仍可能越界。
- `SCPI_ReadEstopTypeQ()` 和 `SCPI_ReadRiskModeQ()` 也把 Flash 中的配置值传给 `SCPI_ChoiceToName()`，虽然 setter 有范围检查，但若 Flash 内容异常，查询路径也最好兜底。

因此，这一条应从“SCPI 参数缺少校验”修正为“硬件读回值、持久化配置值和共享状态值缺少防御式二次范围校验”。

影响:

- SCPI 查询可能读取非法内存。
- 轻则返回异常字符串，重则 HardFault。

建议:

- 对 `mask`、系统状态值、Flash 配置读回值统一做范围检查；`param` 虽由 `SCPI_ParamChoice()` 保护，也可以保留轻量断言作为防御式编程。
- 若超出范围，返回 `"ERR"` 或推送 SCPI error。
- 建议增加 helper:

```c
static const char *ChoiceNameOrErr(scpi_choice_def_t *choices, uint32_t idx, uint32_t max);
```

优先级: P2

说明: 正常用户输入路径已有 `SCPI_ParamChoice()` 保护，风险主要来自异常硬件读回、Flash 内容异常或内存破坏，因此优先级低于初版判断。

### 5. 运行时切换 USART3 波特率破坏 DMA 接收状态

位置:

- `Hardware/libscpi/port/scpi-def.c:267`
- `Hardware/libscpi/port/scpi-def.c:282`
- `Hardware/libscpi/port/scpi-def.c:304`
- `Hardware/libscpi/port/scpi-def.c:314`
- `Core/Src/main.c:227`
- `Core/Src/stm32f1xx_it.c:389`

现象:

`CONFigure:BAUDrate` 命令中:

- 直接修改 `huart3.Init.BaudRate`。
- 调用 `HAL_UART_Init(&huart3)`。
- 使用阻塞 `HAL_UART_Transmit()` 和 `HAL_UART_Receive()`。

但 USART3 正常运行路径是 DMA + IDLE 中断接收:

- 启动时 `HAL_UART_Receive_DMA(&huart3, ...)`
- IDLE 中断中停止 DMA、切换双缓冲、重启 DMA

切波特率命令未停止 DMA、未禁用 IDLE 中断、未与 ModBusTask 互斥。

二次复核:

- 正常 IO 轮询路径已经集中在 `ModBusTask()`，这比 SCPI/状态机直接各自读写 USART3 要好。
- `IO_Read()` 内部也通过 `RS485_WaitRx()` 等待双缓冲 ready 翻转，比盲目 `HAL_Delay()` 更可靠。
- 残余风险主要来自 `CONFigure:BAUDrate` 这类维护命令绕开了 ModBusTask 的事务边界，直接在 SCPI 任务中重初始化 USART3，并混用阻塞收发和 DMA+IDLE 接收。

因此，这不是日常 IO 轮询整体无保护，而是维护命令缺少 USART3 独占事务。

影响:

- 可能破坏 UART3/DMA HAL 状态机。
- ModBusTask 同时读写 RS485 时，帧会互相污染。
- 阻塞接收与 DMA 接收混用，容易出现无法恢复的接收状态。

建议:

- 增加 RS485 总线互斥锁，所有 USART3 操作必须持锁。
- 切换波特率前:
  - 暂停 ModBusTask 或获取独占锁。
  - 停止 UART DMA。
  - 关闭 IDLE 中断。
  - 清 UART 错误/IDLE 标志。
- 切换完成后重新启动 DMA 接收。
- 避免在 SCPI 任务中长时间 `HAL_Delay(1000)`，可改为异步配置状态机。

优先级: P1

## 中风险问题

### 6. UART1 输出路径混用，SCPI 应答和日志可能互相干扰

位置:

- `Hardware/libscpi/port/scpi_port.c:42`
- `Core/Src/usart.c:284`
- `Core/Src/stm32f1xx_it.c:502`
- `Core/Src/stm32f1xx_it.c:504`

现象:

SCPI 输出通过 `printf()`，底层 `fputc/__io_putchar` 使用阻塞 `HAL_UART_Transmit()`。调试日志和 POST 使用 `Uart1_Printf()`，底层走 DMA 非阻塞发送。

两条路径共用 USART1，但没有互斥和统一缓冲。

二次复核:

- `SCPITask()` 对 USART1 接收路径已经做了 `taskENTER_CRITICAL()` 快照保护，本条不涉及接收缓冲一致性。
- `Uart1_Printf()` 用 `Usart1_TX_Flag` 避免 DMA TX 重入，忙时直接丢弃日志；这能降低 DMA 缓冲被并发覆盖的概率。
- 残余问题是 SCPI 应答走 `printf()`/阻塞单字节发送，日志走 `Uart1_Printf()`/DMA，二者共用同一物理 UART，协议输出和日志输出没有统一串行化。

所以本条应表述为“输出通道语义混用”，不是“UART1 完全没有保护”。

影响:

- SCPI 应答与日志交叉输出。
- DMA 正在发送时，阻塞发送可能插入字节或等待异常。
- `Uart1_Printf()` 忙时直接丢弃，关键诊断日志可能丢失。

建议:

- 统一 UART1 输出层。
- 建议建立 TX ring buffer 或 FreeRTOS queue，由一个串口 TX 任务串行发送。
- 区分 SCPI 通道和日志通道，至少在协议应答期间禁止日志插入。

优先级: P1

### 7. RamVector 初始化顺序晚于 ModBusTask 首次使用

位置:

- `Core/Src/main.c:373`
- `Core/Src/main.c:406`
- `Hardware/RamVector/Src/ram_vector.c:21`

现象:

`RamVector_Init(0)` 在 `StateVectorTask()` 内部执行，且该任务启动后先 `osDelay(700)`。`ModBusTask()` 启动后只 `osDelay(500)`，随后读取 `RamVector_GetLocalIO()` 并更新 IO。

虽然 `ram_vector` 是静态变量，默认零初始化，但 magic、end_magic、local_node 等初始化逻辑不是在系统启动早期完成。

二次复核:

- 单机模式下 `local_node` 静态默认值为 0，`node_idx()` 仍返回 0；`ModBusTask()` 首轮读写 `io_state[0]` 通常不会因为未初始化而访问错误位置。
- `ram_vector` 静态零初始化也使命令槽默认为 `VCMD_NONE`，首轮误执行命令的概率不高。
- 真正问题是 `magic/end_magic` 和未来多机 `node_id` 语义未在调度器启动前建立，且 `StateVectorTask()` 内部调用全局初始化会把 ModBusTask 已写入的首帧 IO 清零。

因此，这条不是当前单机必现故障，更像初始化职责边界不清，优先级可降。

影响:

- 首轮 ModBusTask 可能在 RamVector 正式初始化前读写向量表。
- 未来多机模式或 node_id 非 0 时，初始化顺序会变得更危险。

建议:

- 在 `main()` 启动调度器前调用 `RamVector_Init(0)`。
- `StateVectorTask()` 不再负责全局初始化，只负责状态机输入。

优先级: P2

### 8. RTOS 对象创建缺少返回值检查

位置:

- `Core/Src/freertos.c:125`
- `Core/Src/freertos.c:128`
- `Core/Src/freertos.c:137`
- `Core/Src/freertos.c:149`
- `Core/Src/freertos.c:258`

现象:

`osTimerNew()`、`osTimerStart()`、`osThreadNew()` 返回值未检查。若堆不足、配置错误或对象创建失败，系统仍继续运行。

二次复核:

- 问题仍成立，但属于启动失败诊断能力不足，不是已观察到的运行期竞态。
- `SysTimerCallback()` 使用 `ModBusHandle`、`StateVectorHandle` 前没有 NULL 检查，这一点仍值得修。

影响:

- `SysTimerHandle`、任务 handle 可能为 NULL。
- `SysTimerCallback()` 对 NULL handle 调用 `osThreadFlagsSet()`。
- WatchDog 任务创建失败时，系统可能被 IWDG 复位且缺少诊断。

建议:

- 创建每个 RTOS 对象后立即检查。
- 失败时输出错误并进入 `Error_Handler()`，或至少 `configASSERT(handle != NULL)`。

优先级: P1

### 9. 构建配置不一致，存在不存在文件和旧缓存

位置:

- `.eide/eide.yml:44`

现象:

`.eide/eide.yml` 引用:

但该文件当前不存在。

旧构建缓存 `builder.params` 仍包含:

- 删除或修正 `.eide/eide.yml` 中不存在的 `event_table.c`。
- 统一 Keil、EIDE、compile_commands 的源文件清单。
- 明确 `Hardware/BsmRelay/Src/StateMachine.c` 是否完全退出构建。

优先级: P1

### 10. ModBus 事务缺少请求/响应绑定

位置:

- `Hardware/modbus/src/RS485.c:34`
- `Hardware/modbus/src/RS485.c:171`
- `Hardware/BsmRelay/Src/BsmRelay.c:143`

现象:

RS485 接收通过 `Usart3_RX_BUF1_IsReady` 翻转判断“新帧到达”。`IO_Read()` 与 `ReadHoldingRegister()` 都读全局 `Uart3_BuffIsReady`。

当前没有统一事务层来绑定:

- 当前发送的请求。
- 期望的从机地址。
- 期望的功能码。
- 期望的响应长度。
- 响应超时。

影响:

- 连续读输入、读输出时，迟到帧可能被下一次调用消费。
- 异常帧或其他功能码响应可能污染当前读取结果。
- 后续增加更多 ModBus 命令后风险扩大。

建议:

- 建立 `Modbus_Transaction()` API。
- 每次事务内部清旧接收状态、发送请求、等待响应、校验地址/功能码/长度/CRC。
- 所有 RS485 调用统一经过该 API。

优先级: P2

## 可维护性问题

### 11. 旧状态机文件仍保留大量逻辑

位置:

- `Hardware/BsmRelay/Src/StateMachine.c:1`

现象:

文件已标注 `[DEPRECATED]`，但仍包含大量全局变量和状态机逻辑。

影响:

- IDE 搜索和交叉引用结果噪声较大。

建议:

- 或者保留文件但从所有构建、索引、文档入口中显式排除。

优先级: P2

### 12. Flash 保存策略容易消耗擦写寿命

位置:

- `Hardware/libscpi/port/scpi-def.c:121`
- `Hardware/libscpi/port/scpi-def.c:157`
- `Hardware/libscpi/port/scpi-def.c:192`
- `Hardware/libscpi/port/scpi-def.c:227`
- `Hardware/libscpi/port/scpi-def.c:563`
- `Hardware/libscpi/port/scpi-def.c:595`
- `Hardware/Flash/Src/flash.c:311`

现象:

多个 SCPI 配置命令每次执行都调用 `Flash_Save()`，而 `Flash_Save()` 擦写整页。

影响:

- 上位机重复配置相同值也会消耗 Flash 擦写次数。
- 产测或调试脚本如果频繁调用配置命令，会加速配置页磨损。

建议:

- 设置值前比较当前值，未变化则不保存。
- 增加 `SYSTem:CONFig:SAVE` 之类显式保存命令，配置命令只改 RAM 缓存。
- 长期可考虑双页或 wear-leveling。

优先级: P2

### 13. SCPI 参数范围语义不完整

位置:

- `Hardware/libscpi/port/scpi-def.c:244`
- `Hardware/libscpi/port/scpi-def.c:403`
- `Hardware/libscpi/port/scpi-def.c:500`

现象:

部分命令对编号和值缺少明确校验:

- `CONFigure:SWITch#`
- `READ:SWITch#:STATe?`
- `CONFigure:CYLInder#`
- `CONFigure:LED`

非法编号可能静默按默认值或进入 default 分支但仍返回成功。

二次复核:

- `CONFigure:LED` 的颜色值来自 `SCPI_ParamChoice(context, led_source, &param, TRUE)`，枚举值本身已有保护；这里真正缺的是对 `param` 到 `VCMD_LED_OFF + param` 的未来兼容兜底。
- `CONFigure:CYLInder#` 的动作值来自 choice 表，动作值已有保护；但气缸编号除 `2` 外都会落到气缸 1，非法编号不会报错。
- `CONFigure:SWITch#` 和 `READ:SWITch#:STATe?` 的编号范围仍需显式约束。

所以本条应理解为“命令编号/未来扩展语义不完整”，而不是所有 SCPI 枚举参数都未校验。

影响:

- 上位机误以为命令执行成功。
- 问题定位困难。

建议:

- 对所有 SCPI 命令定义参数范围。
- 非法参数返回 SCPI 标准错误，例如 `-222 Data out of range`。
- 对硬件写入失败返回 `-240 Hardware error`。

优先级: P2

### 14. 复位原因采集时机与注释不完全一致

位置:

- `Core/Src/main.c:78`
- `Core/Src/main.c:208`

现象:

注释写“尽早读 RCC->CSR”，实际在 `HAL_Init()` 和 `SystemClock_Config()` 后调用。当前路径下通常仍可工作，但“尽早”的保证不强。

影响:

- 如果后续初始化代码或 HAL 版本变化清除了 reset flags，复位原因会不准确。

建议:

- 若复位原因用于关键诊断，尽量在 `HAL_Init()` 后立即读取，或确认 HAL 不清除 `RCC->CSR` reset flags。

优先级: P3

## 建议修复顺序

1. 修正 IROM 范围，保护最后 2KB Flash 配置页。
2. 修正 `IO_Read()` 长度判断和响应帧校验。
3. 为 `RamVector` 命令槽增加原子 take/clear 机制，保留现有优先级仲裁。
4. 统一 USART3/RS485 维护命令事务，禁止 `CONFigure:BAUDrate` 与 DMA 接收/ModBusTask 并行。
5. 统一 UART1 输出路径，避免 SCPI 应答和日志互相干扰。
6. 清理构建配置，删除不存在文件和旧状态机参与构建的可能性。
7. 增加 RTOS 对象创建失败检查。
8. 为硬件读回值、Flash 配置读回值、系统状态值增加范围保护。
9. 优化 Flash 保存策略。

## 推荐验证项

1. 构建后检查 map 文件，确认最大 RO 地址低于 `0x0803F800`。
2. 构造 6 字节异常 ModBus 响应，验证不会越界访问且返回失败。
3. 压测 SCPI 命令与按钮/急停事件并发，验证命令不会被清槽丢失，且安全优先级仍能覆盖用户命令。
4. 连续发送 `READ:LINK:STATe?`，模拟 GPIO 异常读回；同时篡改 Flash 配置读回值，验证 choice 查询不会越界。
5. 执行 `CONFigure:BAUDrate` 后继续进行 IO 轮询，验证 USART3 DMA 能稳定恢复。
6. 开启调试输出并连续查询 SCPI，验证串口返回不交叉、不丢协议响应。
7. 清理 build 后重新生成交叉引用，确认构建源文件清单唯一且无废弃状态机。
