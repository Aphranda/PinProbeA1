# PinProbeA1 多机版本迁移优化方案

## 文档信息

| 项目 | 内容 |
|------|------|
| 日期 | 2026-06-13 |
| 当前分支 | `feature/can-multinode` |
| 基线版本 | `v0.9.0-standalone` / 稳定单机版 |
| 参考来源 | `archive/main-rough-20260613` 粗糙多机版本、现有 RAM 反射内存方案 |
| 目标 | 在不破坏稳定单机架构的前提下，逐步演进到多机 CAN 协同版本 |

---

## 1. 总体判断

当前单机版本已经形成了比较清晰的工程边界：

- `RamVector` 作为命令、状态、IO 镜像的共享边界。
- `state_vector.c` 负责本机状态机逻辑。
- `cmd_exec.c` 负责把 `VCMD` 命令转换为实际硬件动作。
- SCPI、按键、传感器等输入源只应产生事件或命令，不应绕过执行边界直接改硬件状态。

旧 `main` 中的多机版本有可取的协议设计思路，但实现比较粗糙，不能直接覆盖当前稳定单机版。多机版本应该作为单机版本的扩展层，而不是替换主架构。

---

## 2. 迁移原则

### 2.1 保留现有稳定边界

多机功能必须沿用当前单机版的执行链路：

```text
输入源
  -> RamVector 命令槽
  -> state_vector / ModBusTask 调度
  -> cmd_exec.c
  -> BsmRelay 硬件 IO
```

CAN 或 Modbus 远程命令不能直接调用：

- `Cylinder_Write`
- `Lock_Write`
- `LED_Write`
- 其他 BsmRelay 低层硬件写接口

远程命令进入本机后，也必须转换为 `RamVector_PostLock()`、`RamVector_PostCylinder()`、`RamVector_PostLED()` 等本机命令接口。

### 2.2 多机层只做同步和路由

CAN 协议层只负责：

- 节点发现
- 心跳
- 在线状态
- 远程命令收发
- ACK
- 版本比较
- 状态镜像同步

CAN 协议层不负责：

- 本机状态机决策
- 直接执行气缸、锁、LED
- 覆盖整个 `RAM_Vector_t`
- 在中断上下文中做复杂逻辑

### 2.3 先小闭环，再完整同步

第一版多机功能不追求一次性完成所有内容。优先实现最小可验证闭环：

```text
节点 A 产生远程命令
  -> CAN 广播
  -> 节点 B 接收
  -> 写入节点 B 本地 RamVector 命令槽
  -> 节点 B 本地 cmd_exec 执行
  -> 节点 B 回报 ACK / IO 状态
```

整表同步、快照持久化、复杂冲突仲裁放到后续阶段。

---

## 3. 可复用旧方案内容

旧粗糙多机版本中的以下内容可以吸收：

| 内容 | 处理方式 |
|------|----------|
| CAN 11-bit ID 编码 | 可复用思想：`function + source + destination` |
| CAN 功能码划分 | 可保留 `HEARTBEAT`、`NODE_JOIN`、`ACK`、`DATA_SYNC`、`REG_READ`、`REG_WRITE`、`EMERGENCY` |
| TX 队列 / RX 回调注册 | 可复用模式，但回调中只投递事件，不做重逻辑 |
| 心跳在线检测 | 可复用，建议任务上下文处理超时 |
| 版本号同步 | 可复用，但先用于状态诊断，后续再做冲突处理 |
| 寄存器映射 | 可复用思想，用于 PLC/Modbus 到 RamVector 字段映射 |
| 1024B 向量表分包 | 后期用于掉线重入追赶，不作为第一阶段重点 |
| W25Q128 快照 | 后期持久化阶段再引入 |

---

## 4. 不应直接复制的内容

旧多机版本中的以下实现不应直接复制：

| 内容 | 风险 |
|------|------|
| 旧 `RAM_Vector_t` 的单 `cmd + params` 结构 | 与当前三命令槽结构不兼容 |
| CAN handler 直接写硬件 | 破坏当前执行边界，容易引入状态不一致 |
| ISR 回调中执行复杂逻辑 | 中断时间过长，影响实时性和稳定性 |
| 整表 `memcpy` 覆盖本地向量表 | 可能覆盖本地未处理命令、IO 状态和安全状态 |
| 旧 `StateMachine` 全局变量耦合 | 与当前 `state_vector.c` 架构冲突 |
| 快照功能过早引入 | 增加调试复杂度，不利于先跑通 CAN 闭环 |

---

## 5. 推荐模块分层

### 5.1 `CANCom`：底层收发

职责：

- CAN 初始化封装
- 标准帧 ID 打包 / 解包
- TX 队列
- RX 帧接收
- 过滤器配置
- 基础发送 API

不负责：

- 命令解释
- 节点状态机
- RamVector 写入策略
- 硬件执行

### 5.2 `CANProtocol`：协议处理

职责：

- 解析功能码
- 处理 `HEARTBEAT`
- 处理 `NODE_JOIN`
- 处理 `ACK`
- 处理 `REG_READ / REG_WRITE`
- 处理 `REMOTE_CMD`
- 生成同步事件

要求：

- ISR 或 HAL 回调中只收帧、入队、置标志。
- 协议处理尽量放到任务上下文。
- 不直接调用硬件 IO。

### 5.3 `RamVectorSync`：同步适配层

职责：

- 把远程 CAN 命令转换为当前稳定单机版的 `RamVector_Post*()` 调用。
- 维护远程节点 `node_status[]`。
- 维护远程节点 `io_state[]` 镜像。
- 统一处理版本、ACK、冲突计数。

这是多机功能和现有单机架构之间最关键的适配层。

### 5.4 `cmd_exec.c`：唯一硬件执行入口

职责保持不变：

- 从 `RamVector` 读取命令槽。
- 执行锁、气缸、LED 等实际 IO。
- 执行后清理命令槽。

---

## 6. 命令覆盖规则

当前单机版已经使用三个独立命令槽：

- `cmd_lock`
- `cmd_cylinder`
- `cmd_led`

多机版继续沿用该结构。

建议规则：

1. 同一命令槽内，新命令覆盖旧命令。
2. 普通命令可以覆盖普通命令。
3. 安全命令可以覆盖任何未处理命令。
4. 普通命令不能覆盖未处理的安全命令。
5. 远程命令和本地命令使用同一套优先级规则。
6. 执行层读取命令时，最好一次性取出三个命令槽的快照，再清空已取出的槽，避免执行过程中被新命令打断。

建议保留当前优先级：

```c
#define CMD_PRIO_OBSERVE  0
#define CMD_PRIO_USER     1
#define CMD_PRIO_SAFETY   2
```

## 7. CAN 协议建议

### 7.1 11-bit ID 编码

沿用旧方案思路：

```text
bit10..6 : function  5 bit
bit5..3  : source    3 bit
bit2..0  : dest      3 bit
```

建议约定：

- `source = 1..7` 或 `1..8` 需要根据 3 bit 限制重新确认。
- 若继续支持 8 节点，3 bit 字段只能表达 `0..7`，应明确 `0` 是广播还是 Node8。
- 更稳妥的方案是：`0 = broadcast`，`1..7 = node1..node7`，如果必须 8 节点，需要调整 ID 分配。

此处是多机协议设计中需要优先定稿的问题。

### 7.2 功能码建议

第一阶段建议只实现：

| 功能码 | 用途 |
|--------|------|
| `HEARTBEAT` | 节点在线状态 |
| `NODE_JOIN` | 节点上线广播 |
| `ACK` | 命令或同步确认 |
| `REMOTE_CMD` | 远程命令注入 |
| `IO_STATUS` | IO 状态镜像 |

后续再加入：

| 功能码 | 用途 |
|--------|------|
| `DATA_SYNC` | 关键字段同步 |
| `DATA_REQUEST` | 请求完整向量表 |
| `DATA_RESPONSE` | 分包发送完整向量表 |
| `REG_READ` | 远程寄存器读取 |
| `REG_WRITE` | 远程寄存器写入 |
| `EMERGENCY` | 紧急广播 |

## 8. 阶段计划

### 阶段 0：方案冻结

目标：

- 明确多机分层。
- 明确命令覆盖规则。
- 明确 CAN ID 编码。
- 明确第一版最小闭环。

交付物：

- 本文档。
- CAN 协议字段表。
- RamVector 同步边界说明。

### 阶段 1：CAN 基础通信

目标：

- CAN 初始化。
- CAN 帧收发。
- TX 队列。
- RX 任务处理。
- 心跳广播。
- 在线节点表。

验收：

- 两块板可以互相看到心跳。
- 掉线 1.5s 左右能标记 offline。
- 不影响现有单机控制流程。

### 阶段 2：远程命令注入

目标：

- 节点 A 发送远程命令。
- 节点 B 接收后写入本地 `RamVector` 命令槽。
- 节点 B 仍由本地 `cmd_exec.c` 执行。

验收：

- 远程控制锁、气缸、LED 可用。
- 新命令覆盖旧命令规则生效。
- 安全优先级不被普通命令覆盖。

### 阶段 3：节点状态和 IO 镜像

目标：

- 同步 `node_status[]`。
- 同步 `io_state[]`。
- 支持查询远程节点状态。

验收：

- 任意节点能看到其他节点在线状态。
- 任意节点能看到其他节点主要 IO 状态。

### 阶段 4：ACK、版本和冲突诊断

目标：

- 命令 ACK。
- 版本号递增。
- 版本冲突计数。
- 远程节点未确认诊断。

验收：

- 发起节点可判断哪些节点已执行或已确认。
- 收到旧版本同步时不覆盖本地新状态。

### 阶段 5：完整向量表追赶

目标：

- 掉线节点重新上线后请求完整 1024B 向量表。
- 分包传输。
- CRC / magic 校验。
- 安全地合并远程状态。

验收：

- 节点掉线后重入，可以追上网络状态。
- 不覆盖本地未处理安全命令。

### 阶段 6：W25Q128 快照持久化

目标：

- 保存稳定状态快照。
- 掉电恢复。
- 双区或日志式写入。

验收：

- 掉电恢复后能恢复节点配置和必要状态。
- 快照写入不会阻塞实时控制。

## 9. 第一版最小闭环定义

第一版多机不要做太大，建议只实现以下闭环：

```text
Node A:
  本地产生命令
  -> CAN REMOTE_CMD 广播/单播

Node B:
  CAN 收到 REMOTE_CMD
  -> RamVectorSync 判断 node_mask
  -> RamVector_PostLock / PostCylinder / PostLED
  -> cmd_exec.c 执行
  -> 更新本地 io_state
  -> CAN ACK / IO_STATUS 回报
```

第一版暂不做：

- 完整 1024B 向量表覆盖同步。
- W25Q128 快照。
- PLC 完整 Modbus 从站。
- 多主复杂冲突仲裁。

---

## 10. 关键风险

| 风险 | 应对 |
|------|------|
| 旧方案直接覆盖当前单机架构 | 只提取协议思想，代码重新按当前边界接入 |
| CAN 回调中执行硬件 | 回调只入队，任务上下文写 RamVector |
| 远程命令覆盖安全命令 | 统一命令优先级和覆盖规则 |
| 整表同步覆盖本地状态 | 第一阶段不用整表覆盖，后期做字段级合并 |
| 8 节点与 3-bit ID 字段冲突 | 先定稿 CAN ID 编码，再写协议代码 |
| 快照过早引入导致调试困难 | 快照放到最后阶段 |

---

## 11. 推荐下一步

下一步建议只做设计冻结，不急着改代码：

1. 定稿 CAN ID 编码，尤其是广播和 8 节点编号的关系。
2. 定稿 `REMOTE_CMD` 帧格式。
3. 定稿远程命令到三个命令槽的映射。
4. 定稿 ACK 和 IO_STATUS 的最小字段。
5. 再开始阶段 1 的 CAN 基础通信实现。
